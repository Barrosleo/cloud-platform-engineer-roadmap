# Week 2 — Linux Services, Logs & Networking

This lab demonstrates real operational troubleshooting tasks performed by Cloud, Platform, and Automation Engineers. The focus is on diagnosing failures using processes, logs, networking, and DNS, then documenting findings using a structured engineering workflow.

All work was performed using WSL2 with Ubuntu 22.04, reflecting how Linux is commonly accessed in modern cloud environments.

---

## Why logs matter

Logs are the primary source of truth during incidents. They reveal:

- **Failure details:** What failed.
- **Timeline:** When it failed.
- **Source:** Which component caused the failure.
- **Recurrence:** Whether the issue is recurring.

Effective log analysis reduces downtime and prevents guesswork.

---

## Why networking matters

Most cloud outages are networking-related. Understanding interfaces, routing, ports, and DNS allows engineers to:

- **Isolate issues:** Separate connectivity issues from application failures.
- **Diagnose faster:** Identify outages quickly.
- **Recover safely:** Restore service with minimal impact.

---

## Troubleshooting workflow used

1. **Observe:** Observe symptoms.
2. **Collect:** Collect evidence (logs, commands, outputs).
3. **Identify:** Identify root cause.
4. **Fix:** Apply fix.
5. **Verify:** Verify recovery.
6. **Document:** Document findings.
